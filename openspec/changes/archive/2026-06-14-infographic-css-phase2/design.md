## Context

`infographic.css` exists and covers the six pattern families specified in Phase 1. Three families — `ig-data`, `ig-structure`, and `ig-comparison` — ship with rendering gaps that prevent an LLM from expressing common infographic patterns: a data family that has no visual chart rendering, a structure family that cannot express depth, and a comparison family whose rows are column-local with no way to span traits across options.

All changes are additive. No existing class names are removed or renamed. Existing markup continues to render as before.

## Goals / Non-Goals

**Goals:**
- Make `ig-data-chart` produce visually distinct chart outputs (bar fill, stat grid, donut ring)
- Make `ig-structure` support arbitrary DOM nesting with CSS-only connectors
- Make `ig-comparison` support a trait-spans-columns matrix layout
- Keep every change expressible in static HTML with zero JavaScript
- Update SPEC.md to reflect new elements, variants, and the SVG surface allowance

**Non-Goals:**
- Animated charts or interactive tooltips
- Multi-theme support
- Converter tooling that transforms source documents into HTML
- Changes to `ig-process`, `ig-analysis`, or `ig-code` families
- Any external font or CDN dependency

## Decisions

### D1: `ig-data` uses `style="--ig-data-fill: X%"` for bar width

**Decision**: The proportional bar fill is driven by an inline CSS custom property on each `ig-data-bar` element. The CSS reads `var(--ig-data-fill, 0%)` as the gradient stop.

**Rationale**: An LLM can write `style="--ig-data-fill: 73%"` reliably without needing JavaScript or `width` calculations. The gradient approach (`linear-gradient(to right, accent var(--ig-data-fill), surface 0%)`) keeps the implementation in one CSS rule.

**Alternative considered**: A `width` attribute or inline `width` style on a child element. Rejected because it breaks the two-layer token architecture and leaks layout concerns into HTML.

### D2: `ig-data-chart` requires a variant; no bare default

**Decision**: `ig-data-chart` without a variant is invalid in Phase 2. An LLM must choose `--bar`, `--stat`, or `--donut`.

**Rationale**: The three modes have incompatible internal element sets. Allowing a bare `ig-data-chart` would make the expected children ambiguous to an LLM.

**Alternative considered**: Keep the bare chart as a default that falls back to `--bar` rendering. Rejected because silent fallback makes spec violations invisible.

### D3: `ig-data-chart--donut` uses inline SVG authored by the LLM

**Decision**: The donut/pie mode requires the LLM to write an inline `<svg>` element inside `ig-data-donut`. The SVG uses `stroke-dasharray` on a `<circle>` for the ring. The spec defines the expected SVG structure as a normative example.

**Rationale**: A donut ring is not expressible in pure CSS without `conic-gradient` hacks that have poor LLM reusability. Inline SVG is static HTML — no external dependency, no JS. The `stroke-dasharray` pattern is well-known and reliably generated by LLMs.

**Alternative considered**: `conic-gradient` on a `<div>`. Rejected because the LLM must compute degree values from percentages, which is error-prone without JS.

**Surface change**: Inline SVG is now explicitly permitted as a rendering surface. "CSS-only" is redefined in the spec to mean: zero JavaScript, zero external resources. SVG markup inside the HTML document is allowed.

### D4: `ig-structure` tree nesting uses a `ig-structure-children` wrapper element

**Decision**: Child nodes live inside an `ig-structure-children` div inside their parent `ig-structure-node`. CSS connector lines target `ig-structure-children > .ig-structure-node` with `::before` pseudo-elements.

**Rationale**: DOM nesting is the only approach that allows CSS `::before`/`::after` connectors between siblings at any depth. A `data-depth` attribute cannot be used to draw lines connecting parent to children without JavaScript.

**Alternative considered**: `data-depth` flat list. Rejected because connector lines between nodes cannot be rendered in CSS-only with a flat DOM.

### D5: Flat mode is `ig-structure-tree--flat`; tree mode is the default

**Decision**: The existing flat behavior becomes `ig-structure-tree--flat`. The bare `ig-structure-tree` (current behavior) becomes the tree default. Since all existing examples use the flat visual idiom (indented list, no children wrappers), any existing HTML that did not use `ig-structure-children` can add `--flat` to preserve rendering.

**Rationale**: Tree is the semantically richer default. Flat is the downgrade path.

### D6: `ig-comparison-grid--matrix` is a new opt-in mode; default is unchanged

**Decision**: The existing column-owns-rows model remains the default. The matrix model is opt-in via `ig-comparison-grid--matrix`. The two models have different internal element sets and are not interchangeable.

**Rationale**: The prose comparison (column-owns-rows) is valid for narrative comparisons. The matrix (trait-spans-columns) is valid for structured attribute comparisons. Both appear in real infographics. Forcing one shape would lose expressive range.

**CSS mechanism**: `ig-comparison-grid--matrix` uses `display: grid; grid-template-columns: max-content repeat(var(--ig-col-count, 2), 1fr)`. The LLM sets `style="--ig-col-count: 3"` on the grid element to control column count.

## Risks / Trade-offs

- **SVG authoring errors** → The spec includes a normative SVG example that the LLM can copy verbatim. The `stroke-dasharray` formula is documented so the LLM does not need to compute it from scratch.
- **`ig-structure-children` forgotten** → If an LLM writes nested nodes without the wrapper, nodes will render inline rather than indented. The spec makes `ig-structure-children` mandatory for non-leaf children.
- **`--ig-col-count` mismatch** → If the LLM writes three `ig-comparison-col-head` elements but sets `--ig-col-count: 2`, the grid shifts. The spec requires `--ig-col-count` to equal the number of `ig-comparison-col-head` children.
- **No bar axis or scale labels** → The bar chart has no y-axis, tick marks, or baseline. This is intentional for v1. Complex axis rendering requires JS or significant SVG and is out of scope.
